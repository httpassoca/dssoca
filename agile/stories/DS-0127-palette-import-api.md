## Description

As an app using dssoca, I want to import a custom palette (e.g. generated by the docs Theme
Builder) through the config API or as plain CSS, so I can re-seed the whole design system on a
different accent without forking the stylesheet.

Implementation: `PALETTE_SLOTS` (19 camelCase slots) + `Palette`/`ThemePalette`/`PaletteSlot`
types in `src/lib/dssoca.config.ts`; `DesignConfig.palette: Palette | null` with tri-state partial
semantics in `src/lib/config.ts`; `paletteToCss()` + `paletteSlotVar()`; all barrel-exported.

## Acceptance criteria

- [x] `applyDesignConfig({ palette })` writes the ACTIVE theme's 19 slot values as inline custom
      properties on the target, re-writing on every call so theme flips are never masked.
- [x] `palette: null` clears every inline slot (stylesheet palette resumes); omitting the field
      keeps the current palette (tri-state).
- [x] `getDesignConfig().palette` is a deep copy; `defaultDesignConfig.palette === null`.
- [x] `paletteToCss(palette)` emits `:root,[data-theme='dark']` + `[data-theme='light']` blocks
      with all 19 kebab-cased declarations each (the documented zero-JS/SSR path — load after
      `theme.css`).
- [x] Unit tests cover apply / flip-rewrite / clear / keep / deep-copy / CSS shape; the manifest
      test pins `PALETTE_SLOTS` order.
- [x] `docs/themes.md` gains the "Custom palettes" section (types, runtime semantics, CSS/SSR
      guidance).

## Notes

- Inline custom properties beat both `[data-theme]` blocks regardless of the attribute — hence the
  rewrite-on-every-call design; SSR is deliberately routed through the CSS path instead.
